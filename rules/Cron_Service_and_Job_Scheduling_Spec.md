# ⏱️ Cron Service & Job Scheduling Rules

This document defines the strict architectural rules for implementing scheduled background tasks (Cron Services), configuring execution engines, and tracking execution logs.

---

## 1. Overview & Purpose
The Cron Service is responsible for executing automated, scheduled background tasks. To ensure absolute decoupling, observability, and compatibility with pre-compiled environments, scheduling uses a clean dual-dimension model separating the clock orchestration from the execution payload protocol.

---

## 2. The Dual-Dimension Configuration Model
Every scheduled job in `cron_scheduled_jobs` must be explicitly configured using our standard engine and protocol variables:

*   **`job_type` (The Scheduling Engine - "Who runs the clock")**:
    *   `Cron_Daemon`: Scheduled and managed by our local FastAPI background thread polling cycle.
    *   `PG_CRON`: Delegated entirely to PostgreSQL's native `pg_cron` extension.
*   **`target_type` (The Protocol Medium - "How the task executes")**:
    *   `HTTP_WEBHOOK`: Triggers standard outgoing HTTP network requests.
    *   `ISOLATED_PROCESS`: Spawns a secure, isolated OS subprocess running a Python script under `backend/tasks/*`.
    *   `SQL_FUNCTION`: Runs native PostgreSQL PL/pgSQL database procedures directly.

### Valid Combinations Mapping
1. `job_type = 'Cron_Daemon'` ➡️ `target_type IN ('HTTP_WEBHOOK', 'ISOLATED_PROCESS')`
2. `job_type = 'PG_CRON'` ➡️ `target_type IN ('HTTP_WEBHOOK', 'SQL_FUNCTION')`

---

## 3. Database Schema Design

The scheduling system requires two primary tables: `cron_scheduled_jobs` (configuration) and `cron_job_execution_logs` (history).

### 3.1. `cron_scheduled_jobs` Table
```sql
CREATE TABLE cron_scheduled_jobs (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    job_name VARCHAR(255) NOT NULL UNIQUE,
    description TEXT,

    -- Scheduling
    cron_expression VARCHAR(50) NOT NULL, -- e.g., '0 * * * *' (Hourly)
    is_active BOOLEAN DEFAULT TRUE NOT NULL,

    -- Execution Details
    job_type VARCHAR(50) DEFAULT 'Cron_Daemon' NOT NULL, -- 'Cron_Daemon', 'PG_CRON'
    target_type VARCHAR(50) DEFAULT 'HTTP_WEBHOOK' NOT NULL, -- 'HTTP_WEBHOOK', 'SQL_FUNCTION', 'ISOLATED_PROCESS'
    target_payload JSONB DEFAULT '{}'::jsonb NOT NULL, -- Stores URLs, headers, SQL commands, or modules

    -- Standard Requirements
    version INTEGER DEFAULT 1 NOT NULL,
    created_by UUID REFERENCES auth.users(id),
    updated_by UUID REFERENCES auth.users(id),
    created_at TIMESTAMPTZ DEFAULT now() NOT NULL,
    updated_at TIMESTAMPTZ DEFAULT now() NOT NULL,
    is_deleted BOOLEAN DEFAULT FALSE NOT NULL,
    deleted_at TIMESTAMPTZ,

    CONSTRAINT chk_job_type CHECK (job_type IN ('Cron_Daemon', 'PG_CRON')),
    CONSTRAINT chk_target_type CHECK (target_type IN ('HTTP_WEBHOOK', 'SQL_FUNCTION', 'ISOLATED_PROCESS'))
);
```

### 3.2. `cron_job_execution_logs` Table
*   **IMPORTANT:** Because this table name contains the word `log`, it is **EXEMPT** from standard audit triggers to prevent recursive logging loops.
```sql
CREATE TABLE cron_job_execution_logs (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    job_id BIGINT NOT NULL REFERENCES cron_scheduled_jobs(id),

    status VARCHAR(50) NOT NULL, -- 'RUNNING', 'SUCCESS', 'FAILED'
    started_at TIMESTAMPTZ DEFAULT now() NOT NULL,
    completed_at TIMESTAMPTZ,

    duration_ms BIGINT,
    error_message TEXT,
    response_payload JSONB,

    created_at TIMESTAMPTZ DEFAULT now() NOT NULL,
    updated_at TIMESTAMPTZ DEFAULT now() NOT NULL
);
```

## 4. Secure Subprocess Guidelines (`ISOLATED_PROCESS`)
When creating standalone Python tasks to execute via the `Cron_Daemon` as an `ISOLATED_PROCESS`:
1.  **Strict Storage**: The task file MUST be placed directly inside `backend/tasks/` (e.g. `backend/tasks/custom_task.py`).
2.  **No Dynamic Imports**: NEVER import task files dynamically into the main FastAPI thread. They must run in their own subprocess to isolate memory, prevent crash propagation, and remain persistent across container rebuilds.
3.  **Namespace Constraints**: For security (**CWE-94 / CWE-95**), paths are strictly sanitized. Subprocesses are blocked unless they match a safe module name within the `backend.tasks.*` package.
4.  **The Stdout JSON Protocol**: Independent scripts must print diagnostic logs to standard output normally, and output their final metrics on a single line starting with the prefix `__CRON_RESULT__ ` containing the standard JSON structure:
    ```json
    {
      "status": "SUCCESS",
      "message": "Human readable status summary",
      "duration_ms": 420,
      "response_payload": { "items_processed": 50 },
      "error_message": null
    }
    ```

5.  **Workspace Safety Protocol (Concurrent Agents):** Any autonomous AI agent executed via cron tasks MUST strictly adhere to `@rules/Workspace_Safety_Protocol.md`. The cron task wrapper MUST forcefully inject a system instruction reminding the AI agent to never run destructive git commands (e.g., `git restore`, `git clean`, `git reset`) that could discard untracked or uncommitted changes made by concurrent user activity or other AI agents.

## 5. Execution Logging Protocol
Whenever a task is triggered, the runner MUST:
1.  **START:** Insert a record into `cron_job_execution_logs` with `status = 'RUNNING'` and `started_at = now()`.
2.  **EXECUTE:** Run the target logic.
3.  **COMPLETE/FAIL:** Update the exact same `cron_job_execution_logs` record:
    *   Set `completed_at = now()`.
    *   Calculate and set `duration_ms`.
    *   Set `status` to `'SUCCESS'` or `'FAILED'`.
    *   If it failed, capture the exception trace in `error_message`.

### 5.1. Asynchronous Webhook Background Tasks Cleanup Protocol
When an `HTTP_WEBHOOK` endpoint defers execution to a FastAPI `BackgroundTasks` function (for long-running operations), the HTTP endpoint returns `{"status": "accepted"}` immediately.

To prevent the dashboard UI from displaying the job as perpetually `Running`, the deferred background task function **MUST** implement a `finally:` block that cleans up the global scheduler state and logs the execution end.

#### Code Pattern for Asynchronous Webhook Tasks:
```python
import time
from backend.server.state import global_state

def run_async_workflow(log_id: str) -> None:
    start_time = time.time()
    try:
        # 1. Execute core workflow logic here
        ...
    except Exception as e:
        logger.error(f"Task failed: {e}")
    finally:
        # 2. Retrieve active background jobs set
        duration_ms = int((time.time() - start_time) * 1000)
        active_jobs = getattr(global_state, "active_background_jobs", set())

        # 3. Identify and remove this job name from active background jobs
        job_name = "My Task Name"
        found_job = None
        for name in active_jobs:
            if "my_task_identifier" in name.lower():
                found_job = name
                break

        if found_job:
            job_name = found_job
            try:
                active_jobs.remove(found_job)
            except KeyError:
                pass

        # 4. Notify scheduler to reset state to Idle
        scheduler = getattr(global_state, "scheduler", None)
        if scheduler:
            scheduler._update_task_state(job_name, "Idle")
            
            # 5. Log execution completion to DB logs if log_id is valid
            parsed_log_id = None
            if isinstance(log_id, int):
                parsed_log_id = log_id
            elif isinstance(log_id, str) and log_id.isdigit():
                parsed_log_id = int(log_id)
                
            if parsed_log_id is not None:
                scheduler._log_execution_end(
                    parsed_log_id,
                    "SUCCESS",
                    duration_ms,
                    response_payload={"message": "Task completed successfully."}
                )
```

## 6. Direct Database Access Rules & Client Integration (Self-Supabase)

To enable seamless autonomous management of scheduled jobs by AI agents without cluttering backend codebases with static programmatic startup-seeding blocks, agents are authorized to directly read and update job schedules in the `self-supabase` database.

### 6.1 Strict Tooling Requirement (CRITICAL)
**Rule:** To add, update, or remove a cron job schedule, you MUST NOT interact with the database directly using raw SQL queries or generic database tools. You MUST call the `register_cron_job` tool provided in your toolset (DevOpsTools).

This tool automatically validates the schema, safely updates the `cron_scheduled_jobs` table via upsert, and importantly, forces the frontend UI and backend scheduler to synchronize instantly.

### 6.2 The Registry Synchronization Concept
Because the FastAPI background `Cron_Daemon` thread holds an in-memory task registry (`self._dynamic_jobs`) that populates the admin UI, direct database modifications (such as direct inserts) are normally invisible. The `register_cron_job` tool automatically solves this by hitting the `POST /api/admin/cron/reload` endpoint internally after the database upsert completes.
